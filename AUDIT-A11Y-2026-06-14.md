# ♿ Creator Hub — Accessibility & ADHD-friendly audit (2026-06-14)

A whole-hub pass focused on **accessibility (WCAG 2.2 AA)** and **ADHD / cognitive
design** (W3C COGA + GOV.UK). Grounded in a sourced rubric (bottom of this doc), then
five parallel auditors swept every page for: keyboard/focus, form labels & names,
heading/landmark structure, ADHD/cognitive patterns, and motion/contrast/target-size.

Same methodology as the prior audits: **every finding was re-verified against the
actual source before anything was touched**, only self-contained additive changes were
shipped, the riskier/high-volume ones are written up with exact code, and rejected
false-positives are listed so they don't get "re-fixed" later.

The hub started from a strong base — five a11y modes (dark / dyslexia / tint / low-stim
/ focus), global `prefers-reduced-motion`, focus-ring baseline, `EggieUI` helpers. So
this pass is mostly about **gaps in that system** and per-page regressions. Two headline
gaps: the a11y modes were only reachable from the home page, and there was no skip
link / `main` landmark anywhere.

## ✅ Shipped this pass (safe, additive, verified)

### Systemic — one change in a shared file, reaches every page

1. **A11y modes reachable on every page** *(biggest ADHD/access win)* — the mode pills
   (🌙📖🟡🧘🎯) lived only in the home page's top-utility row, so anyone deep-linking to
   a tool page couldn't turn them on. `hub-nav.js` now renders an **Accessibility**
   section at the bottom of the ☰ menu (present on every page), filled via a new
   `HubA11y.refresh()` hook in `a11y-modes.js`. Gated on `a11y-modes.js` being present,
   so pages without it (the OBS overlay) are unaffected. *(WCAG 3.2.3; COGA "easy to
   find settings".)*
2. **Skip link + `main` landmark on every page** — `hub-core.js` now injects a
   "Skip to content" link as the first focusable element and marks the page's first
   `.wrap` as `role="main"` (reusing an existing `<main>`/`[role=main]` if present). The
   skip target is the page's `<h1>` (real content start, past the nav, which sits inside
   `.wrap`), falling back to the landmark. Styles in `hub-ui.css` (`.ui-skip`,
   off-screen until focused). *(WCAG 2.4.1 Bypass Blocks, 1.3.1.)*

### Per-page (calmer files only — see "deferred" for the concurrently-edited ones)

3. **Keyboard-operable "Sign in" pill** — `ask.html`, `review.html`, `finance.html` had
   `<div class="auth-pill" onclick>` (not keyboard reachable). Added
   `role="button" tabindex="0"` + an Enter/Space `onkeydown`. *(WCAG 2.1.1.)*
4. **Keyboard-operable prompt chips** — `ask.html`'s 11 `<span class="chip" onclick>`
   suggestions got a small upgrade script (role/tabindex + Enter/Space → `quick()`).
   *(WCAG 2.1.1.)*
5. **Focus ring on editable output** — `sponsor-pitch.html` `.out-output[contenteditable]`
   had `outline:none` and no replacement; added a `:focus-visible` ring. *(WCAG 2.4.7.)*
6. **Contrast: Media Kit CTA card** — `media-kit.html` `.cta-card` was white text on a
   light pastel gradient (~1.4:1). Switched to deep brand ink (same call as hub-ui.css
   pass-3). *(WCAG 1.4.3.)*
7. **Tap targets ≥ 24px** — `finance.html` `.modal .x`, `todo.html` `.te-pill .x`,
   `niche-quiz.html` `.recipe .combo .x` were ~12–18px. Added
   `min-width/height:24px` + flex centering. *(WCAG 2.5.8.)*
8. **One clear `<h1>` per page** — `todo.html` (`<span class="logo>` → `<h1>`),
   `sponsor-pitch.html` (`<div class="logo>` → `<h1>`), `manager-claim.html` (the
   always-visible brand logo demoted from `<h1>` to `<div class="logo">` so the active
   view's `<h1>` is the only one). `.logo` is class-styled, so all three look identical.
   *(WCAG 1.3.1 / 2.4.6.)*
9. **Form labels / names** — added `aria-label` + `autocomplete="email"` to the
   placeholder-only email/question inputs on `ask.html` (`#q`, `#emailInput`,
   `#modalEmail`) and `review.html` (`#emailInput`, `#modalEmail`). *(WCAG 1.3.1, 1.3.5.)*
10. **Undoable delete** — `brand-memory.html` series/links/facts ✕ buttons deleted
    typed-out content in one click; wrapped each in a `confirm()` (matches the page's
    existing sign-out confirm). *(COGA "easy to undo errors" / WCAG 3.3.4.)*
11. **Descriptive link text** — `manager-claim.html` "click here" → "open the planner".
    *(GOV.UK.)*

**Cache-bust:** `a11y-modes.js?v=6→7`, `hub-core.js?v=5→6`, `hub-nav.js?v=9→10` bumped
across all pages; `hub-ui.css?v=3→4` via hub-core's injector.

**Files changed:** `hub-ui.css`, `hub-core.js`, `hub-nav.js`, `a11y-modes.js` (shared);
`ask.html`, `review.html`, `finance.html`, `sponsor-pitch.html`, `media-kit.html`,
`todo.html`, `niche-quiz.html`, `brand-memory.html`, `manager-claim.html`; all `*.html`
(version bump); `changelog.json`.

## 🟠 Recommended — verified, NOT blind-shipped (exact code below)

Deferred because they're either high-volume, need a visual/demo-mode check, or live in
files the perf audit flagged as **actively edited by parallel sessions**
(`analyzer.html`, `planner.html`). Each is real and verified.

### 1. The big one: labels missing `for=` (WCAG 1.3.1 / 4.1.2) — ~130 controls
`media-kit.html` (~45), `finance.html` (17), `review.html` (8), `sponsor-pitch.html`
(~10), `brand-memory.html` (~11) write `<label>Text</label><input id="…">` as visual
siblings with **no `for=`** — looks labeled, zero programmatic association. The inputs
already have `id`s, so the fix is purely additive: add `for="<id>"` to each preceding
label. High value, mechanical, but should be done per-form with a quick check that each
label maps to the right `id` (best done in-editor against the live form, not blind).
Representative lines per file are in the audit notes; e.g. `review.html` review
textareas at 363/369/375/381/387/393/399/411.

### 2. `analyzer.html` (off-limits — concurrent edits; apply in a quiet window)
- **Contrast (1.4.3):** `.impact-badge.low`/`.high` (`:194-197`) and `.mark.warn`
  (`:209-211`) use white text on light fills (~1.5–1.8:1). Fix: badge/glyph text to
  `color:var(--ink)` / `#5a3a00` for warn, or use full-saturation fills.
- **Names (4.1.2):** `#deleteTemplateBtn` 🗑 (`:1457`) → `aria-label="Delete template"`;
  JS sticky-chip ✕ (`:8036`) → `title="Remove"` (peers already have it).
- **Target size (2.5.8):** remove-button cluster `.eb-sticky-chip .rm` (`:7272`),
  `.eb-candidate-chip .rm` (`:7359`), `.syn-rm` (`:7342`) → `min-width/height:24px`.

### 3. `planner.html` (huge + concurrent — apply carefully)
- **Contrast (1.4.3):** `.tw-cat` (`:1777`) white text on `var(--cat-color, rgba(144,165,255,0.2))`
  fails when the category colour is unset/light. Same issue as `todo.html`
  `.tc-cat-chip` (`:454`). Robust fix is luminance-aware text colour per category; quick
  fix is a darker fallback default. Left for a focused pass since user-set light colours
  also fail — worth solving once, properly, for both pages.
- **Alt text (1.1.1):** JS `<img alt="${m.name||''}">` (`:3797`) / `${b.label||''}`
  (`:5375`) become `alt=""` when empty though informative → non-empty fallback
  (`'Attached media'` / `'Brand image'`).
- **Names:** several JS-template inputs + `#twInput`/`#searchInput`/`#tweetSearchInput`/
  invite-link inputs need `aria-label`.

### 4. `sponsor-pitch.html` — autosave the pitch draft (COGA "avoid data loss")
The multi-step pitch form (`pf_*` fields) is only held in memory until an explicit cloud
save — an accidental reload loses everything typed. `analyzer.html`/`review.html` already
autosave; mirror that pattern:
```js
// debounced localStorage draft of the pf_* fields, restored on load when no pitch is open
const PF_DRAFT='sp_pitch_draft';
const pfFields=['pf_name','pf_brand_name','pf_brand_url','pf_brand_desc','pf_audience_fit','pf_personal_angle','pf_pricing','pf_status','pf_outcome_notes'];
let pfT; document.addEventListener('input',e=>{ if(e.target.closest('#pitchForm')){ clearTimeout(pfT); pfT=setTimeout(()=>{ const d={}; pfFields.forEach(id=>{const el=document.getElementById(id); if(el)d[id]=el.value;}); localStorage.setItem(PF_DRAFT,JSON.stringify(d)); },400); }});
// on load, if not editing an existing pitch: JSON.parse(localStorage.getItem(PF_DRAFT)) → fill fields; clear on successful savePitch()
```
Medium risk (touches form flow) — test in demo mode.

### 5. Smaller, safe-to-apply-anytime
- **Headings (1.3.1, P3):** `niche-quiz.html` result cards jump h2→h4 (make `.ax-name`
  an `<h3>`); `faq.html` in-answer `<h4>` labels skip h3 (→ `<h3>`, keep size via CSS);
  `media-kit.html` sign-in view shows an `<h2>` with no `<h1>` (make `:498` the `<h1>`
  or add a visually-hidden one).
- **Names (P1/P2):** `aria-label` on placeholder-only inputs in `toolbox.html` (4
  inputs), `growth.html` `#playbookSearchInput`, `debut-checklist.html` `#debutDate`,
  `finance.html` filter selects + 6 modal `×` close buttons (`aria-label="Close"`),
  `planner-editor.html` `.pc-editor-notes`.
- **`autocomplete` tokens (1.3.5, P3):** add `autocomplete="email"` to the remaining
  email inputs (finance `:413`, sponsor-pitch `:727`); `autocomplete="name"` to
  media-kit `:718`, planner invite-name inputs.
- **Walls of text (COGA, P3):** split the longest paragraphs in `growth.html` (esp.
  `:1154`, 627 chars) and optionally `faq.html` `:391`.
- **Nicer confirms (presentational):** the native `confirm()` dialogs (analyzer,
  planner, todo, habits, sponsor-pitch…) are correct but could adopt
  `EggieUI.modal.confirm()` from `ui-modal.js` for a consistent, on-brand dialog. Low
  risk; do it as each page is next touched (and add `<script src="ui-modal.js?v=1">`).

## 🧯 Verified and REJECTED (don't re-flag)

- **Motion / sound:** clean. No `<audio>`, no `<video autoplay>`, no JS `.play()`. All
  looping animations are pure CSS (gated by the global reduce-motion + low-stim rule);
  JS loops are user-initiated (teleprompter has play/pause), real-time clocks (exempt
  under WCAG 2.2.2), or already reduce-motion-guarded (analyzer score count-up).
- **ADHD P1 (catastrophic irreversible actions):** none — every task/template/comment
  delete is already `confirm()`-guarded; analyzer/review autosave; the tab-refocus guard
  protects unsaved edits.
- **Color-only signaling (1.4.1):** essentially clean — every status pill/dot/mark
  carries a text label or symbol (status word, ✓/!/✗, +/− sign, ⭐ priority star).
- **Focus rings:** page-level `input:focus{outline:none}` rules are **not** failures —
  the 0-specificity `:where(...):focus-visible{box-shadow:var(--ui-ring)}` baseline in
  hub-ui.css still renders the ring. (Only the `contenteditable` case, not in that
  selector list, was a real gap — fixed above.)
- **Positive tabindex:** none anywhere. **Keyboard traps:** none — the analyzer
  combobox is a correctly-built ARIA widget (arrow/Enter/Escape, outside-click close).
- Inline editor "Remove row" splices on not-yet-saved in-memory lists (media-kit,
  sponsor-pitch deliverables, todo subtasks) — recoverable by re-adding; not flagged.
  (brand-memory was different — it `persist()`s immediately.)

## 📋 Rubric & sources (what we audited against)

WCAG 2.2 AA highlights: 1.4.3 text contrast 4.5:1 / 1.4.11 non-text 3:1; 2.4.7 focus
visible; 2.4.11 focus not obscured; 2.5.8 target size 24px; 2.1.1 keyboard; 2.4.1 skip
link; 1.3.1 structure/labels/landmarks; 4.1.2 name/role/value; 4.1.3 status messages;
2.3.3 + `prefers-reduced-motion`; 1.4.10 reflow; 1.4.12 text spacing; 3.1.1 lang.
ADHD / COGA + GOV.UK: undoable errors; persist in-progress work; "Step X of Y";
feedback on long operations; chunk text (no walls); limit competing CTAs; descriptive
controls; consistent layout; minimize interruptions; calm visuals; plain language.

Sources: W3C WCAG 2.2 (`w3.org/TR/WCAG22/`) + Understanding docs; W3C COGA "Making
Content Usable" (`w3.org/TR/coga-usable/`); GOV.UK "Dos and don'ts on designing for
accessibility"; WebAIM WCAG 2 checklist; Deque WCAG 2.2 updates.
